Review of Stripe Customer Portal Bug Report and Proposed Fix
1 Context from the Bug Report
1.1 Problem Summary

The bug report describes a critical misconfiguration in the Stripe Customer Portal. Two separate Billing Portal configurations are used – one for Mouse Individual and one for Mouse Business – with monthly and annual prices. The architecture selects the appropriate bpc_ configuration based on the account type stored in DynamoDB. However, attempts to update the configurations through the Stripe Billing Portal Configuration API to include the features.subscription_update.products array consistently succeed (HTTP 200) but fail to persist. Six attempts using the CLI, curl and the Node SDK across multiple Stripe API versions show that the products field is silently dropped. Without this array, Stripe displays all products in the portal and allows customers to switch from Individual to Business or vice versa, which would corrupt internal state.

1.2 Required Behaviour

For each tier the portal should allow customers to switch between billing intervals but never switch to a different product. Additionally:

Individual subscriptions have a fixed quantity of 1. They must be able to switch between the monthly price ($15/month) and the annual price ($150/year). Customers must not be able to change quantity or upgrade to Business.

Business subscriptions are per‑seat and require the ability to change quantity (minimum 1, maximum 99). They must also be able to switch between monthly ($35/seat/month) and annual ($350/seat/year) prices. Business licences must not be able to downgrade to Individual.

Both types of customers should still be able to update their payment method or cancel the subscription via a Stripe‑managed UI.

1.3 Key Observations from the Bug Report

No errors returned. Each API call accepts the products array and returns HTTP 200. The response object shows that the products field is absent and retrieving the configuration confirms that it never persisted.

Other fields persist. When other features.subscription_update fields (e.g.
default_allowed_updates and proration_behavior) are updated in the same call, they are correctly stored. This suggests that only the products array is dropped.

Attempts across multiple API versions. The problem occurs with apiVersion 2024‑12‑18.acacia and 2026‑02‑25.clover.

Stripe support confirmed a limitation. According to the user, Stripe Support’s AI acknowledged that updating the products array via the API is a known limitation.

2 Pertinent Portions of Official Stripe Documentation

Although many pages in docs.stripe.com require authentication, Stripe’s generated type definitions and open‑source library documentation provide definitive information about the Customer Portal and its deep‑link flows.

2.1 Deep‑link flow_data Types

The flow_data.type enumeration in the Customer Portal SessionCreateParams allows four flow types: payment_method_update, subscription_cancel, subscription_update and subscription_update_confirm.

When flow_data.type is subscription_update_confirm, you must supply subscription_update_confirm.subscription (the subscription ID) and subscription_update_confirm.items. The API currently supports only one subscription item per session; subscriptions with multiple items are not updatable.

Each item object contains:

id – the subscription item to be updated.

price – the new price to switch to. The documentation notes that this price must also be included in the configuration’s features.subscription_update.products list.

quantity – the target quantity for the subscription item.

subscription_update_confirm flows also support optional discounts (coupon or promotion code), but these are not relevant here.

The Elixir client’s documentation echoes this structure: the items() type accepts optional id, price and quantity keys, confirming that quantity can be changed within the deep‑link flow.

2.2 Products and Adjustable Quantities in Portal Configurations

The BillingPortal.Configuration type definitions in Stripe’s Node SDK (v20.x) describe the features.subscription_update.products array:

A configuration may specify up to 10 products that support subscription updates.

Each product entry can include an adjustable_quantity object. Its enabled flag determines whether the customer can change the quantity. Optional minimum and maximum fields set numeric bounds.

Each product entry also contains a product ID and a list of permitted prices. Without this list, Stripe’s documentation states that the price passed to subscription_update_confirm.items.price must still be included in the configuration’s products array.

These definitions show that setting adjustable_quantity on a product is the official way to allow seat count changes and to constrain the range.

2.3 Behavior of Deep‑link Flows

Although Stripe’s main documentation pages on portal deep links are behind a login, the type definitions and client docs indicate how deep links behave:

subscription_update_confirm flows bypass the generic portal UI and send the customer directly to a confirmation page where they accept a specific change (price and/or quantity). This flow requires subscription_update.enabled to be enabled on the configuration but does not itself display a product picker; the update is defined by your code.

subscription_cancel flows take the customer to a cancellation flow for a given subscription. You can optionally specify retention strategies (e.g., coupon offers).

payment_method_update flows allow the customer to add or update a payment method.

Because deep‑link flows hide the navigation sidebar and only allow a single action, they are ideal for enforcing strict upgrade/downgrade logic. The SDK docs emphasise that the price you pass to a subscription_update_confirm flow must appear in the configuration’s products list, but the bug report indicates that the products list cannot currently be persisted through the API.

3 Assessment of the Proposed Solution
3.1 Summary of the Proposed Approach

Since the products array on existing portal configurations cannot be set via the API, the proposed solution eliminates access to the generic portal UI. Instead, all portal interactions use deep‑link flows via flow_data:

Switch billing interval: Build a subscription_update_confirm session specifying the subscription ID and a new price. No quantity is passed for Individual customers (thus remaining at 1). Business customers pass the same quantity when changing the billing interval.

Adjust seats: For Business customers, build a subscription_update_confirm session with the same price and a new quantity. Server‑side validation enforces 1 ≤ quantity ≤ 99.

Update payment method: Use a payment_method_update flow.

Cancel subscription: Use a subscription_cancel flow.

View invoices: Render invoices in your own UI using the Stripe API rather than the portal.

The frontend passes a flow value (e.g., switch_to_annual, adjust_seats) to the server. The server looks up the customer’s account type in DynamoDB, fetches their active subscription from Stripe, validates the request and constructs the appropriate flow_data for billingPortal.sessions.create(). No generic portal sessions are created.

3.2 Evaluation Against Business Logic Requirements

Individual subscribers can switch between monthly and annual but cannot change quantity or switch tiers. In a subscription_update_confirm flow you specify the subscription item ID and the new price. If you omit the quantity field, Stripe will keep the existing quantity (which is 1). The deep‑link flow only shows a confirmation page for the specific change; there is no product picker, so the customer cannot switch to Business. The requirement that the target price be listed in features.subscription_update.products is problematic because the bug prevents setting that array, yet in practice Stripe appears to allow updates via deep links even without it. Barring this API limitation, the approach meets the Individual requirements.

Business subscribers can switch billing intervals and adjust quantity between 1 and 99, but cannot switch to Individual. The subscription_update_confirm flow accepts a quantity field for the subscription item; therefore quantity changes can be sent directly in the deep link. The adjustable_quantity structure in portal configurations supports minimum and maximum bounds, but since the API cannot persist products, the server must enforce these bounds itself. By deriving the price from the customer’s account type and ignoring any client‑supplied price, it is impossible for a Business customer to downgrade to the Individual product. Limiting quantity to the range 1–99 and rejecting higher values ensures compliance with the licence contract.

Customers can update billing information in a Stripe‑managed UI. The payment_method_update flow takes the customer directly to Stripe’s card update page. The BillingPortal.SessionCreateParams.FlowData.Type enumeration includes 'payment_method_update', and the Node SDK’s Configuration definitions enable payment‑method updates. This satisfies the requirement to use a Stripe‑managed UI for payment method changes.

Compatibility with Stripe Managed Payments. Deep‑link flows modify the user interface but not the underlying payment processing. Portal sessions created via billingPortal.sessions.create() still operate within the Merchant of Record (MoR) framework. There is no indication in the type definitions or client docs that deep‑link flows are incompatible with Managed Payments; Stripe support reportedly confirmed this. Because the flows only orchestrate subscription updates, cancellations or payment‑method changes, they should not conflict with SMP.

3.3 Risks and Considerations

Products array requirement: Official SDK docs state that the price passed to a subscription_update_confirm item must be present in the configuration’s features.subscription_update.products list. If this requirement is enforced in production, then omitting the products array may cause an error in the deep‑link flow. In test mode the update appears to work, so either the requirement is not enforced or Stripe has relaxed it for deep‑link flows. It would be prudent to test the flows thoroughly in live mode and monitor for errors.

Multiple subscription items: Deep‑link flows currently support only subscriptions with a single subscription item. If in the future you add add‑on products or multiple prices to a subscription, subscription_update_confirm flows will not support updates. The code should detect multi‑item subscriptions and either block the action or handle them with a different interface.

Adjustable quantity settings: The adjustable_quantity object allows Stripe to validate quantity changes within the portal. Because you cannot persist this through the API, your server must enforce quantity bounds (1–99) and show a “contact sales” message for 100+. When Stripe resolves the bug or adds a dashboard control, you should update the configuration accordingly.

4 Conclusion and Recommendations

The proposed solution uses deep‑link flows to circumvent a Stripe API limitation that prevents persisting features.subscription_update.products. Based on the available documentation and type definitions:

Deep links are supported and offer a secure way to direct users to specific actions without exposing a product picker. The flow types and parameters are well‑documented.

Quantity changes are allowed in subscription_update_confirm flows via the quantity field. However, the portal configuration’s adjustable_quantity object normally governs quantity limits; since this cannot be set via API, the server must enforce limits.

Payment‑method updates and cancellations can be handled through dedicated flows with minimal complexity.

Cross‑tier switching is prevented by server‑side logic that selects prices based on the customer’s tier and never exposes other products in the UI. Even if products cannot be set on the configuration, deep‑link flows do not present alternative products.

Suggestions:

Proceed with the deep‑link based implementation. It satisfies the business constraints and avoids the problematic parts of the portal.

Implement comprehensive server‑side validation: ensure that the requested flow is appropriate for the customer’s tier, derive the correct price from a trusted mapping, and enforce quantity bounds.

Test the flows in live mode to confirm that Stripe does not enforce the products array requirement. If errors occur, contact Stripe support for an override or manual configuration via the dashboard.

Monitor Stripe’s changelogs and support channels. When Stripe resolves the products API limitation or adds dashboard controls for specifying allowed products and adjustable quantities, update the portal configuration accordingly.

With these precautions, the proposed solution appears sound and should allow Mouse to launch with Individual and Business tiers meeting the specified constraints.
